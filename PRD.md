> 版本：v0.4-draft · 日期：2026-05-10 · 作者：PM

## Goal Anchor

| 项目 | 内容 |
|------|------|
| **北极星** | 让分享出去的文档链接，对接收者来说"一眼专业、信息完整、阅读舒适" |
| **必做** | ① 文档 meta 栏展示创建时间 + 文档 ID；② HTML 内容渲染；③ 宽度上限改为 120rem（≈1200px） |

## 第一性原理拆解

### 真问题是什么？

用户说"新增一个更规范的文档详情页"，但本质上有三个独立问题：

1. **信息缺失**：现有 `doc.html` 的 meta 栏只有文档 URL，没有创建时间 —— 接收链接的人不知道这是新文档还是旧文档。
2. **HTML 渲染**：需求说"Markdown 渲染为 HTML 展示"—— 这是现在 **已经实现的**（goldmark 渲染，`.markdown-body` 展示），本质上是**确认现状而非新功能**。
3. **宽度太窄**：`.doc-page` 当前 `width: min(88rem, ...)` ，对于长表格、宽代码块来说阅读体验差，改为 120rem 是合理的样式调整。

### 真假约束识别

| 约束 | 类型 | 结论 |
|------|------|------|
| 需要显示创建时间 | 真约束 | 后端已存，需要模板传参 |
| "新页面"而非改现有页面 | 假约束 | 改 `doc.html` + `style.css` 即可，不需要新路由 |
| HTML 渲染 | 现状已满足 | 无需额外开发，文档确认即可 |
| 1200px | 真约束（用户明确） | 一行 CSS 改动 |

## MVP 边界（三栏）

### ✅ 必做

1. **meta 栏新增创建时间**：`doc.html` 中展示 `CreatedAt`（格式：`2026-05-10 14:32 UTC`），后端模板传参需确认字段。
2. **meta 栏保留文档 ID**：现有 `fastmd.dev/{{.ID}}` 保留，样式不变。
3. **HTML 渲染确认**：现有 goldmark 渲染已满足，doc.html 的 `.markdown-body` 已正确展示，**不需要新开发，文档注明即可**。
4. **宽度改为 120rem**：`style.css` 中 `.doc-page` 的 `width: min(88rem, ...)` 改为 `min(120rem, ...)`。

### ❌ 不做（含理由）

1. **新增路由/新建页面文件** — 改现有 `doc.html` 即可，新路由带来额外维护成本（分类：过度工程）
2. **显示文档字节数/字数** — 存储层未记录，需要改 store，成本高于价值（分类：成本不匹配）
3. **编辑/更新文档功能** — 超出本次范围，属于独立需求（分类：范围蔓延）
4. **代码高亮主题切换** — 无需求驱动，属于锦上添花（分类：锦上添花）

### ⏸ 暂缓

1. **文档作者/来源展示** — 多租户场景下有意义，但当前 token 体系不对外展示身份
2. **移动端 meta 栏折叠优化** — 现有响应式已处理 `flex-direction: column`，可优化但不阻塞
3. **目录（TOC）侧边栏** — 长文档有价值，但需要 goldmark 扩展支持，独立排期

## 被砍掉的诱惑

> 用户原话有"新增一个更规范的文档详情/展示页面，不同于现有的简单渲染页"——这暗示可能要**新建一个独立页面/路由**（如 `/doc/:id`）。

**砍掉理由**：现有 `GET /:id` 路由 + `doc.html` 模板完全可以承载所有增强内容，新路由会造成：① 旧链接兼容问题；② 两套模板维护；③ SEO canonical 混乱。本次只增强现有页面即可，"规范"体现在内容和样式，不体现在 URL 结构。

## BDD 验收标准

### 场景 1（正常）：查看文档页面显示创建时间

```gherkin
Given 一篇文档已通过 CLI/API 推送，文档 ID 为 "abc12345"
When 用户在浏览器访问 https://fastmd.dev/abc12345
Then 页面 meta 栏中显示该文档的创建时间，格式为 "YYYY-MM-DD HH:mm UTC"
And 创建时间内容不为空、不为 "0001-01-01"
```

### 场景 2（正常）：文档 ID 保留展示

```gherkin
Given 同上，文档 ID 为 "abc12345"
When 用户访问 https://fastmd.dev/abc12345
Then meta 栏中可见文字 "fastmd.dev/abc12345"（monospace 字体）
And "Raw" 和 "Copy link" 按钮均可见且可点击
```

### 场景 3（正常）：Markdown 渲染为 HTML

```gherkin
Given 推送内容包含 "# Hello" 和 "**bold**" 和 "`code`"
When 用户访问文档页面
Then 页面中可见 <h1> 标签包含 "Hello"
And <strong> 标签包含 "bold"
And <code> 标签包含 "code"
And 所有内容在 .markdown-body 容器内
```

### 场景 4（正常）：宽度改为 1200px

```gherkin
Given 用户在桌面浏览器（viewport ≥ 1400px）访问文档页
When 页面完全加载
Then .doc-page 元素的计算宽度等于 1200px（120rem，基准 10px）
And 内容区不因宽度增大而出现水平滚动条
```

### 场景 5（边界）：旧文档（4 位 ID）兼容

```gherkin
Given 存在一篇旧格式 4 位 ID 文档 "ab12"
When 用户访问 https://fastmd.dev/ab12
Then 页面正常渲染，meta 栏 ID 显示为 "fastmd.dev/ab12"
And 若后端未记录创建时间，则时间栏显示 "—" 或隐藏，不显示 "0001-01-01"
```

### 场景 6（异常）：不存在的文档 ID

```gherkin
Given 文档 ID "zzzzzzzz" 不存在
When 用户访问 https://fastmd.dev/zzzzzzzz
Then 返回 404 页面，不崩溃
And 404 页面样式与其他页面一致（使用 base.html）
```

### 场景 7（响应式）：移动端 meta 栏

```gherkin
Given viewport 宽度 ≤ 780px
When 用户访问任意文档页面
Then meta 栏垂直堆叠（flex-direction: column）
And ID、创建时间、操作按钮均可见，无内容被截断
```

## 技术实现要点（供 RD 参考，不做规定）

> PM 不做技术方案，以下仅为帮助 RD 快速定位的上下文：

- **创建时间字段**：需确认 `store.go` 中 Document 结构体是否有 `CreatedAt time.Time`，以及模板 handler 是否已传递该字段。
- **模板修改范围**：仅 `web/templates/doc.html`（加 `CreatedAt`）+ `web/static/style.css`（改 `.doc-page` 宽度）。
- **宽度数值**：`min(120rem, calc(100% - 3.2rem))`，`.container` 本身已是 `min(120rem, ...)` 一致性良好。
- **时间格式建议**：`2006-01-02 15:04 UTC`（Go time format）。
- **兼容旧 ID**：路由逻辑已有，无需改动。
